Api\Controller\Hotel\SearchController
Class SearchController offers programmatic access to our search functionality. A hotel search is a search for hotels within one city for a defined period of time for a fixed number of rooms. A search can have two kinds of states, 'running' and 'finished'. When a search is started, its state is 'running' and it has 0 results. As providers reply, the number of results increases. As soon as all providers replied, the search status becomes 'finished'. Each result has a limited lifetime since the providers only lock an offer for a certain time. As soon as a result expires, its search expires. Searches can be restarted without having at hand their parameters, the ID is sufficient. Each result contains the price for the accommodation, information about breakfast and amenities as well as the URL of a thumbnail. It also contains a booking URL which points to our user interface for booking. Alternatively, a different booking interface can be implemented using the Hotel Booking API. Use the booking_api_url to get a document that describes what data the user has to provide. If more detailed information about a result is needed, it can be access via a result URL. This URL points to detailed descriptions, collections of image URLs and ratings from different providers. Caching The offers that are returned by the API can expire, so we discourage caching on the client side. To minimize response time and network traffic the API supports HTTP Etags and datestamps. If your network library has Etag and datestamp support, we recommend using If-None-Match- or If-Modified-Since-Headers. If one of the clients Etag matches the resource on the server or the resource was not updated since the last request, it will send an empty reply with a 304 Not Modified status. If you have both options of using If-None-Match or If-Modified-Since, we recommend using If-None-Match (Etags). Abbreviations M Mandatory O Optional @package Api\Controller\Hotel

Start a Search To start a search, you need following information: A location (city, poi or hotel), an arrival date, a departure date and information about how many rooms of which type are requested (see below). The city_id is typically obtained using a call to /locations/cities/name/{query}.{format} (an autocompleter). For different types of searches, the results differ to some extend (see below). The normal flow is to perform a city-based search, let the user pick a hotel, then start a hotel-based search and let the user pick a rate. City-based search or poi-based search: These types of searches return approximate prices and availabilities and the rate-element will be null. When using booking URLs from results of those searches, the booking might not be possible in some cases. Hotel-based search: This type of search returns up-to-date prices and availabilities and the rate element will contain information about the type of room, the cancellation policies and the original currencies and price before the conversion to Euros. Also the payment type is returned, it can have the value 'prepaid' meaning the customer has to pay in advance or 'hotel' meaning the customer has to pay at the hotel. There are five types of locations you can search in. Exactly one has to be specified: city_id Integer The city ID as obtained from the location API poi_id Integer The poi ID as obtained from the location API hotel_id Integer The hotel ID as obtained from the results of another search hotel_ids String Comma separated list of Hotel IDs latitude and longitude Float The geo coordinates to create a Search. The default radius is 20 There are three ways to define a room configuration: 'room_configuration' String 'single' or 'double' or: 'single_rooms_count' Integer Number of single rooms 'double_rooms_count' Integer Number of double rooms or: If you want to search with children you can pass a string of the following format: Rooms are enclosed in brackets, guests are separated by pipes, 'A' represents an adult guest, an integer represents a child of that specific age. Example: '[A|A|3][A|12|14]' means two rooms, the first with one adult and a child of age 3, the second room with a single adult and two children, age 12 and 14. 'room_configuration' String Room configuration with children Please note that the total number of rooms can not be greater than 3. Special case for POSTing with XML as parameters If you need to supply the POST parameters as a XML document in the POST body (Details and important notes here), there is a caveat: You need to supply an enclosing tag, "search", in order for the request parameters to be recognized. Natural language and currencies Check24 serves the German speaking market. Thus, all descriptions are in German and all prices are in Euro. In contrast, all machine-readable aspects are in English, like URLs, parameter names and tag names. Error messages are in German, so they can be displayed without translation. The programmatic handling of errors should rely on the HTTP status codes. Formats The API supports XML and JSON. The preferred format is specified with the last part of the URL: '.json' or '.xml' JSONP The API supports JSONP. Whenever a client requests s resource in JSON format and passes a 'callback' parameter, the reply is wrapped into a function call with that name. When using JSONP, the http status will always be 200, so the browser is inserting the reply into the DOM. Errors can be identified by having only the key 'errors'. Example: When the data in the reply is a hash '{"hello": "there"}' and the 'callback' parameter is set to 'my_funct', the reply will be 'my_funct({"hello": "there"})'.
